## What Went Well

### 1. Strong-prior source inspection saved a full spike worth of work

Story 8-1 confirmed Plan A from `~/.cargo/registry/.../async-opcua-server-0.17.1/src/node_manager/memory/simple.rs:180-262` source-grep alone — the spike code that ran empirically did so to *pin* what source already proved. **Zero `src/` changes** for subscription support.

The lesson: when validating an architectural assumption against a third-party library, audit the library source first, then write the spike test to pin the audit's conclusion. Don't write the spike to *discover* the conclusion; the source already has it.

### 2. The library-wrap-not-fork pattern is now project-shaping

Three Epics in a row have used the same shape:

- Epic 7 (sprint-status 7-2): `OpcgwAuthManager` implements `AuthManager` rather than forking async-opcua's auth path.
- Epic 7 (sprint-status 7-3): `AtLimitAcceptLayer` is a `tracing-subscriber::Layer` parsing a pre-existing event rather than forking the session manager.
- Epic 8 (Story 8-3): `OpcgwHistoryNodeManager` wraps `SimpleNodeManagerImpl` rather than forking the node manager.

Each one bridges a missing async-opcua callback in-process. Cleaner than maintaining a fork; cleaner than waiting for upstream. The brittleness of each (async-opcua emission style for `AtLimitAcceptLayer`, trait surface enumeration for `OpcgwHistoryNodeManager`) is documented in `deferred-work.md` and tracked at upstream FRs (#94).

### 3. `[storage].retention_days` extension over spec's parallel field — design improvement caught at implementation time

Story 8-3's spec called for a new `[storage].history_retention_days` field. Implementation extended the existing `retention_days` field (already feeding the prune loop) instead. **One source of truth instead of two**, no risk of the prune loop and the HistoryRead path disagreeing about retention. Documented as a field-shape divergence from spec; user accepted at iter-1 review (D2).

The lesson generalises: spec field-shape decisions made before implementation can be revisited if implementation surfaces a strictly cleaner design. The CLAUDE.md scope-discipline rule does NOT mean "implement the spec verbatim" — it means "implement what the spec asked for, no more". A simpler field shape that closes the same AC is still in scope.

### 4. The unfiltered-`DataChangeFilter` discovery (Story 8-2 AC#3.3) is a load-bearing security finding documented honestly

`async-opcua 0.17.1::MonitoredItem::notify_data_value:514-517` dedupes on `value.value` only when `FilterType::None` (no filter supplied). A non-compliant SCADA client that omits the filter would silently freeze on the last-good value during a ChirpStack outage — exactly when the operator most needs the alarm signal.

Story 8-2 documented this **as the contract** (compliant clients pass a filter; the OPC UA Part 4 §7.17.2 default `trigger: StatusValue` routes through `is_changed()` and detects status transitions). Pinned by AC#3.3 test. **Not** silently fixed; **not** documented as "fixable later"; **explicitly** documented as the wire-level contract clients must comply with.

This is the right move for a Phase B gateway: clients are FUXA / Ignition / UaExpert, all of which supply the filter. A future SCADA vendor that omits it gets a deterministic failure mode (frozen value), not a silent partial degradation. The honest documentation is more valuable than a server-side workaround that would mask the upstream library behaviour.

### 5. Test count growth was honest and integration-test-heavy

581 (Epic 7 baseline) → 592 (Story 8-1) → 641 (Story 8-2) → 702+ (Story 8-3). Every increment reflects production-driven coverage:

- 8-1: +9 integration tests (2020 LOC of test code) for the subscription contract.
- 8-2: +49 tests (validation + cross-knob + ChirpStack outage + advanced knob plumbing).
- 8-3: +59 tests (storage method + HistoryRead handler + retention upsert).

**No synthetic tests** slipped through Epic 8 reviews. The class of issue Story 8-3 caught — `BadUserAccessDenied` blocking the override before it runs because `HISTORY_READ` access level wasn't set — is exactly the kind only integration tests find. Edge Case Hunter + Acceptance Auditor both flagged it; the combined coverage is what makes the catch reliable.

### 6. The deferral discipline held — operator-action items are tracked, not silently dropped

Three operator-action items deferred during Epic 8 implementation, **all tracked at GitHub issues**:

- #93 — manual FUXA / Ignition / UaExpert verification (single integration pass post-Epic-9)
- #94 — upstream FR for session-rejected callback + subscription-rejection callbacks
- #95 — 5-minute `--load-probe` run for spike report § 7

Plus #98 covering the four out-of-scope HistoryRead extensions (`HistoryReadProcessed` / `AtTime` / per-metric retention / dynamic retention reload).

None of these are "TODO comments in code" (which are easy to lose). Every one is on GitHub with a concrete trigger condition for re-prioritisation. The Epic 7 retro lesson "carry-forward debt is real" continues to be respected — and now operator-action items have their own bucket distinct from dev-work debt.

## What Was Hard

### 1. Story 8-2 was committed as a single combined "Code Review Complete" commit, violating the per-story commit cadence

The CLAUDE.md "BMad Workflow Commit & Push Discipline" rule says:

> After implementing a story (status flips in-progress → review): create a commit with the story's deliverables.
> After a code review (status flips review → done): create a follow-up commit capturing the review fixes.
> Each commit covers exactly one story (or one retrospective).

Story 8-1 (single commit `72201d4`) is OK — a spike doesn't need a separate review commit.

Story 8-3 (commits `0d1ea37` + `efc0383`) followed the cadence cleanly.

**Story 8-2 (commit `cb206d6` only) did not.** Implementation + iter-1 fixes + iter-2 fixes all rolled into one "Code Review Complete" commit. This breaks per-story per-iteration rollback (we cannot `git revert` to "Story 8-2 implementation, before review fixes" because that state never existed as a commit). It also makes the retrospective harder — the diff doesn't show what was added by review.

**Why it happened:** the CLAUDE.md rule explicitly mentions "implementation" and "review" as separate commits, but doesn't mention the case where the review iterates twice. The author conflated "review iterations are within review" with "review is one commit total" and bundled.

**Action item below:** clarify the CLAUDE.md rule. Per-iteration commits or per-story-end-of-review commits are both acceptable; what's NOT acceptable is rolling implementation in with review.

### 2. Library field-name asymmetries cost time

`epics.md:682, 724` says `max_monitored_items_per_subscription`. async-opcua 0.17.1's `Limits` field is `max_monitored_items_per_sub`.

Story 8-2's spec dutifully copied the longer form. Implementation surfaced the divergence at the `serde` deserialisation step — figment couldn't auto-map without an explicit `#[serde(rename)]` — at which point the choice was: rename the library or rename the spec. The library wins (one source of truth; downstream Rust consumers see the same name).

**Cost:** ~half an iteration of redundant naming churn across spec, code, validation, tests, and docs. The spec itself remained as evidence of the asymmetry; deferred-work entry records the rationale.

**Lesson:** when spec drafting names a field that a third-party library also names, `cargo doc --open` the library and copy-paste the field name. Don't paraphrase. The `_per_sub` vs `_per_subscription` divergence cost more than the four characters saved.

### 3. The 8-4 alarm-condition gap is a real product gap, not a story we forgot

Story 8-4 (threshold-based alarm conditions) was always tagged as backlog after Epic 7's Phase A carry-forward analysis flagged it as Phase B work. The Epic 8 plan deferred it. The user confirmed in this retrospective that it stays deferred.

**But:** SCADA operators who deploy this gateway expecting FR23 (threshold-based status-code propagation) will see "no alarm" instead of "no support for alarms" — a silent gap. The fix is not to *implement* 8-4 (out of scope for Epic 8); it's to make the absence **observable** to SCADA users via the Story 6-3 known-failures diagnostics path. See § Known Failures below.

**Cost not paid:** had we not classified 8-4 as KF, an operator deploying for the first soil-moisture-low-alarm scenario would have spent debugging time discovering the gap. KF tagging makes the discovery happen at deploy-config time, not runtime.

### 4. Test-harness duplication crossed the 4th-file threshold — and was deferred anyway

CLAUDE.md scope-discipline rule says "duplicate when third-user threshold isn't crossed". Story 8-3 created the **fourth** integration-test file with `setup_test_server` / `pick_free_port` / `build_client` / `init_test_subscriber` overlap (`tests/opcua_history.rs` joins `tests/opcua_subscription_spike.rs` + `tests/opc_ua_security_endpoints.rs` + `tests/opc_ua_connection_limit.rs`).

The threshold was crossed; extraction to `tests/common/mod.rs` would have been compliant. **It was deferred** because the four files diverge in subtle ways:

- `init_test_subscriber` requirements differ (spike-tests use private `tracing-test::internal::global_buf`; auth-tests use the same; history-tests don't need it).
- TestUser fixtures differ across stories (security stories use `default-user`; subscription/history use `test-user`).
- Per-test-file `serial_test::serial` annotations differ (the spike-tests need it for tracing-subscriber composition; history-tests don't).

A clean extraction would either flatten these differences (forcing one shape) or expose all of them as parameters (over-engineering for 4 callers). The deferred-work entry records the rationale.

**Cost:** the next integration-test file (likely from Epic 9 web-UI testing) will face the same fork-decision. Without a clean extraction story, the duplication will continue to grow. **Action item below:** open a defined cleanup story for `tests/common/mod.rs` extraction with explicit handling of the divergence — should land before the 5th integration-test file appears.

### 5. Pre-existing storage flake `test_concurrent_write_read_isolation` surfaced under heavy parallel load

When `cargo test --lib --bins --tests` runs the full workspace concurrently, `src/storage/sqlite::tests::test_concurrent_write_read_isolation` can return `Err(SQLITE_BUSY)` for all 50 reads and `found == 0` triggers the assertion. Passes deterministically in isolation.

**Not** caused by Epic 8 changes. Pre-existing race assumption. Story 8-2 surfaced it; Story 8-3 didn't reproduce because the writer/reader race window is small and load-dependent.

**Fix path:** mark `#[serial_test::serial]` to remove parallel-load contention, OR rewrite the test to wait for at least one write to complete before starting reads. The 1-line `#[serial_test::serial]` fix is the smaller patch.

**Why deferred:** outside Epic 8 scope per CLAUDE.md scope-discipline. **Action item below:** address before Epic 9 starts (Epic 9 will add Axum web-server integration tests; the workspace-level concurrent test load grows further).

### 6. Doctest baseline (56 failures) carried forward — fourth epic in a row

Epic 4 retrospective: carry forward.
Epic 5 retrospective: carry forward.
Epic 6 retrospective: escalate to defined story.
Epic 7 retrospective: defined story but not yet opened; "promote from carry-forward to a defined story" was an action item.

The compounding entropy is real: every new doc-comment example added across Epics 6/7/8 has had to coexist with a 56-failure baseline, masking any new genuine doctest regression among the noise.

**Action item below:** doctest cleanup is now **blocking before Epic 9 starts**. No more carry-forward. Either open `9-X-doctest-cleanup` as a defined Epic 9 prep story, or schedule an explicit `epic-A-consolidation` between Epic 8 and Epic 9.

### 7. NodeId collision when two devices share `metric_name` is a pre-existing latent bug Epic 8 surfaced

`src/opc_ua.rs:~834-870`: `node_to_metric.insert(...)` (and likely `add_variables(...)`) silently overwrites when device A and device B both have a metric named "Moisture", since NodeId is constructed from the metric_name only. HistoryRead would silently return data from one device under a NodeId conflating two.

**Pre-existing latent bug** (not introduced by Story 8-3). Epic 8's HistoryRead path made it observable: a SCADA client querying device-A's "Moisture" history could receive rows that originated from device-B's "Moisture".

**Action item below:** open a defined GitHub issue for this. **Severity: HIGH** — silent data corruption on multi-device deployments. The fix likely needs the NodeId to include `device_id` (`ns=2;s=device-{id}/{metric_name}` or similar). Schema migration NOT required (NodeId is server-internal); but address-space changes are observable to existing SCADA configurations and should land before Epic 9 ships any operator-facing feature that depends on stable NodeIds.

## Known Failures (KF) — operator-visible feature gaps

Story 8-4 (threshold-based alarm conditions) is **not implemented** in Epic 8. Per the user's Phase B planning, it is deferred to the Phase B backlog.

To prevent silent expectation drift between deployed gateway behaviour and operator expectations, **Story 8-4 is classified as a Known Failure** following the project's Story 6-3 (remote diagnostics for known failures) convention.

**KF disclosure:**

- **What's missing:** FR23 (threshold-based alarm conditions). The gateway does NOT propagate `Bad` / `Warning` status codes when a metric value crosses a configured `low_alarm` / `high_alarm` threshold. SCADA clients see `Good` status for all metric reads regardless of threshold proximity.
- **What still works:** Story 5-2's stale-data status codes (`Uncertain` after 1× poll interval, `Bad` after 3× poll interval) function unchanged. Story 8-2's `DataChangeFilter`-driven subscriptions deliver these status transitions correctly.
- **Workaround for SCADA configurations:** define alarm thresholds in the SCADA application (FUXA / Ignition) rather than in opcgw. This is the operationally sound choice anyway — alarm-condition state belongs in the SCADA client which has full context of acknowledged / suppressed / shelved state, not in a stateless gateway.
- **Rationale for deferral:** Phase B's load-bearing functionality is real-time data (FR21 — done) + historical access (FR22 — done) + web configuration (Epic 9). Threshold alarms are a value-add tier that compositions cleanly into the SCADA-client tooling that already handles alarm acknowledgement state machines.
- **Trigger to revisit:** an operator surfaces a deployment scenario where threshold-driven status-code propagation cannot be moved to the SCADA client (e.g., legacy SCADA stack without local alarm support, or compliance constraint requiring server-side alarm provenance). At that point, open a defined story in Phase B backlog that's NOT named "8-4-threshold-based-alarm-conditions" — the original 8-4 spec was scoped for the wrong epic phase and should be redrafted with the lessons from Stories 8-2 / 8-3 (specifically: the `DataChangeFilter` `trigger: StatusValue` path is the correct integration point).

**Tracking:**

- `deferred-work.md` "Story 8-4 (Threshold-Based Alarm Conditions, Known Failure 2026-05-01)" entry — to be added by the retrospective commit.
- `sprint-status.yaml` keeps `8-4-threshold-based-alarm-conditions: backlog` (no schema change needed; the KF classification lives in deferred-work.md).
- A future `docs/known-failures.md` consolidating this and any other KF entries — currently no such file; deferred until KF count >2.

## Lessons Learned

1. **Audit library source first; then write spikes to pin the audit, not to discover.** Story 8-1's spike confirmed Plan A from grep alone; the empirical run pinned the audit's conclusion. **Don't write a spike to discover what `cargo doc --open` already shows.** Generalises to any third-party library integration.

2. **The library-wrap-not-fork pattern is now load-bearing.** Three Epics, three uses, three success cases. Future stories that need to bridge a missing async-opcua callback should default to this pattern (composition + targeted override) and only escalate to upstream FR if the wrap can't be made testable. The Epic 7 retro lesson "Layer-as-control-plane is brittle" still applies — the brittleness is documented in deferred-work, not silently accepted.

3. **Spec field-shape decisions can be revisited at implementation time if a strictly cleaner shape emerges.** Story 8-3's `[storage].retention_days` extension over the spec's parallel `history_retention_days` field saved a source-of-truth duplication. ## Significant Discoveries

### Discovery 1: Access-control is enforced *before* node-manager overrides run

Story 8-3 implementation initially set up `OpcgwHistoryNodeManager` with the `history_read_raw_modified` override, expected it to fire on HistoryRead requests, and saw `BadUserAccessDenied` returned to the client instead. **The override was never reached.** async-opcua's session dispatch checks the variable's access-level mask and the historizing flag *before* routing the call to the node manager.

**Fix:** metric variables now carry `HISTORY_READ` access level + `historizing = true` at registration time in `OpcUa::add_nodes`. Without this, the override is dead code.

**Impact:** affects every future story that adds a node-manager-side override (e.g., `history_update`, `query` services). The pre-condition is the access-level / historizing flags on the variable, NOT the node manager registration. This belongs in the Epic 9 architecture notes.

### Discovery 2: NodeId metric-name-only construction is a multi-device data-corruption vector

Pre-existing in `src/opc_ua.rs:~834-870`: `node_to_metric.insert(...)` keyed by metric_name only. Two devices with identical metric names share a NodeId. Reads from one device return data from the other (whichever was registered last). HistoryRead from Story 8-3 made the corruption observable — a SCADA client browses to "Device A → Moisture" and gets historical rows that include device B's data.

**Severity: HIGH** for multi-device deployments. Single-device deployments are unaffected. Test fixtures use single-device-per-app shapes which is why the bug never surfaced before.

**Mitigation:** opcgw deployments today have a single device per application most of the time (per the shipped `config/config.example.toml`). The bug is latent until a multi-device deployment lands. Epic 9 (web configuration) will explicitly enable multi-device shapes — fixing the NodeId construction before Epic 9 ships is structural.

**Action item below.**

### Discovery 3: Spike-test productionisation is overdue

Story 8-2 review iter-1 enumerated 7 distinct findings against the test infrastructure inherited from Story 8-1's spike — `clear_captured_buffer` swallowing mutex errors, `set_global_default` failure not surfaced, `open_session_held` returning `None` for any failure, 300ms sleep insufficient on loaded CI, `HeldSession::Drop` calling `event_handle.abort()` without await, etc. All deferred to "a future spike-test productionisation pass".

**The pass was never opened.** The findings carry forward into Epic 9's expanded test surface, where they will continue to mask flakes and silently pass tests under broken instrumentation.

**Action item below:** open a defined cleanup story for spike-test productionisation. Estimate: small (~1-2 days; mechanical replacements). Trigger: before Epic 9 starts.
