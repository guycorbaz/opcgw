## Executive Summary

Epic 7 closed Phase A's last security gap. Three stories shipped:

- **Story 7-1** stripped real credentials from `config/config.toml`, introduced `REPLACE_ME_WITH_*` placeholder rejection in `AppConfig::validate`, and added hand-written redacting `Debug` impls on `ChirpstackPollerConfig` / `OpcUaConfig` (centralised `REDACTED_PLACEHOLDER` / `PLACEHOLDER_PREFIX` constants in `src/utils.rs`). The shipped template now boots into a clear placeholder error rather than booting with leaked secrets. Operator workflow is `cp .env.example .env` + edit; Compose `:?` guards fail-fast on missing host env. Tonic 0.14.5 metadata-leak audit was clean, so no `EnvFilter` mitigation was needed today (follow-up #82 captures the proactive `tower::Layer` strip-authorization-header work for when `TraceLayer` adoption becomes a realistic risk).
- **Story 7-2** introduced `OpcgwAuthManager` (`src/opc_ua_auth.rs`) implementing `async_opcua::server::AuthManager` with **HMAC-SHA-256-keyed credential digests** for fully-constant-time comparison (no length oracle) and sanitised-username `event="opcua_auth_failed"` audit logging. NFR12 source-IP audit is satisfied via **two-event correlation** between async-opcua's `Accept new connection from {addr}` info event and our auth-failed warn — the `AuthManager` trait does not pass peer `SocketAddr` to `authenticate_username_identity_token`, so the upstream feature request is tracked at GitHub issue #85 / Story-7-2 deferral. New `src/security.rs` module ships `validate_private_key_permissions` (NFR9: `0o600` file + `0o700` parent dir enforced at startup; both-violation accumulation; fail-closed) and `ensure_pki_directories` (FR45: race-free auto-create via atomic `DirBuilder::mode()`). Three review iterations triaged 21 patches + the HMAC refactor; 10 deferred entries logged.
- **Story 7-3** added `[opcua].max_connections: Option<usize>` (default `10`, hard cap `4096`) wired through `ServerBuilder::max_sessions`, plus a session-count `info!` gauge (5s tick) and an `AtLimitAcceptLayer` tracing-`Layer` that emits `warn!(event="opcua_session_count_at_limit", source_ip, limit, current)` correlated against async-opcua's accept event. The Layer-as-control-plane pattern is new for this project — first time we've used a `tracing-subscriber::Layer` to bridge a missing upstream callback.

Production blockers: zero. Phase A security goals from the PRD (FR42, FR44, FR45, NFR7, NFR8, NFR9, NFR12, NFR24) are satisfied. Eight follow-up GitHub issues remain open (#82–#90); none block Epic 8 entry. Two **silent-degradation findings** from the retrospective discussion need attention before Epic 8 amplifies the attack surface: NFR12 source-IP correlation breaks under `OPCGW_LOG_LEVEL=warn` (decided action item below: emit a one-shot startup warn), and per-IP rate limiting (#88) becomes materially more important once subscription clients can spawn long-lived multi-monitored-item sessions.

## What Went Well

### 1. Per-story commit + per-story-review commit cadence held cleanly

Epic 6's retrospective made per-story commits a CLAUDE.md rule directly in response to the 6-1/6-2/6-3 entanglement. Epic 7 followed the rule from Story 7-1: six commits total (three implementation + three code-review), each story isolated, per-story rollback intact. This is the first epic where the rule was load-bearing from day one. No exception requests.

### 2. The HMAC refactor is the cleanest part of the auth manager

Pass-3 of Story 7-2's review surfaced a non-constant-time `username == self.user && password == self.pass` short-circuit comparison. The pass-3 patch replaced this with HMAC-SHA-256-keyed digests over both fields, compared with `subtle::ConstantTimeEq`. Net: ~35 LOC, no length oracle, defense-in-depth `is_empty` short-circuit. The user's stated preference ("prefer one pass too much instead of missing issues") explicitly endorsed this iteration.

The lesson generalises: when a security review finds a pattern that *could* leak by timing, the constant-time fix usually clarifies the production code's semantic intent (digests-over-secrets is a stronger contract than equality-over-strings). Helper extraction in Epic 6, HMAC refactor in Epic 7 — both are review-driven refactors that *improve* readability.

### 3. Three rules from Epic 6 stayed load-bearing — none needed amendment

- **Per-story commit discipline**: held throughout (6 commits, no bundling).
- **Loop-until-LOW code review**: Story 7-2 went three iterations; no story flipped to `done` with open HIGH/MEDIUM. Loop terminated under condition #1 (zero unresolved) for 7-1 and 7-3, condition #2 (only LOW remaining after explicit-accept) for 7-2's 10 deferrals.
- **Documentation Sync**: every story landed `docs/security.md` increments + Planning row updates in the same commit. README never drifted.

These are now project-shaping; Epic 8 inherits them without modification.

### 4. Two-event NFR12 correlation pattern proven reusable

Story 7-2 introduced the pattern (auth-failed warn correlated against async-opcua's accept info-event). Story 7-3 reused it for connection-cap-rejection events via `AtLimitAcceptLayer`. **Reuse is the lesson** — when the upstream library doesn't expose a callback, parse a pre-existing event and re-emit at our preferred level. Cleaner than maintaining a fork; cleaner than a polling layer.

The pattern's known weakness (silent degradation under WARN-only log levels) is the action item below.

### 5. Test count growth was honest, not synthetic

488 → 520 → 554 → 581 across the epic. Every increment reflects new production-driven coverage:

## What Was Hard

### 1. async-opcua's `AuthManager` trait doesn't expose peer `SocketAddr`

NFR12 explicitly requires source IP in failed-auth audit logs. `async_opcua::server::AuthManager::authenticate_username_identity_token` receives `&str username, &SecureChannel password, &str endpoint` — no peer address. Story 7-2 spent a meaningful chunk of design budget validating this gap (audited the trait against async-opcua 0.17.1's local source via `cargo metadata`-resolved manifest path) before settling on the two-event correlation pattern.

**Cost:** the correlation breaks silently under `OPCGW_LOG_LEVEL=warn` (see action item below). Not visible until an incident-response operator working from filtered logs notices the source-IP field is empty.

### 2. Three review iterations on Story 7-2 was time-expensive

Each pass: triage of 30+ raw findings into patches / decisions / deferrals, then implementation, then re-review. Pass-2 caught regressions from pass-1 patches; pass-3 caught new findings the deeper inspection surfaced (HMAC weakness, NFC vs NFD username normalisation, Trojan-source Unicode). Net duration: roughly equivalent to Stories 7-1 + 7-3 combined.

**Why it happened:** Story 7-2 has the largest blast radius — `OpcgwAuthManager` is the gatekeeper for every OPC UA session — and the credential-comparison logic was where the project crossed from "redact secrets in logs" (7-1) to "defend against active credential probing" (7-2). The HMAC refactor was load-bearing for the production threat model.

**Per the user's explicit preference**, this is *not* framed as a quality concern. "Prefer one pass too much instead of missing issues" is the stated rule. The review-iteration count is not a metric to minimise. Memory updated with this preference.

### 3. The `tracing-test::internal::global_buf()` private API workaround

Story 7-2's integration tests use `tracing_test::internal::global_buf()` to bypass `traced_test`'s scope filter (the auth events run in async-opcua's spawned-task span, outside the test function's name-matched scope). This is a private API; pinning `tracing-test = "=0.2.6"` would close the brittleness, but the spec didn't require it. Deferred-work entry recorded.

**Hidden cost:** if a `tracing-test` patch release renames the buffer, integration tests break with a confusing compile-error rather than a logic-error. Caveat documented in spec Dev Notes.

### 4. The Layer-as-control-plane pattern is brittle to async-opcua emission style

Story 7-3's `AtLimitAcceptLayer` parses the message field of async-opcua's `Accept new connection from {addr}` event by `starts_with("Accept new connection from ")` + IP extraction. If a future async-opcua release switches to structured `message = "..."` field-style emission or quote-wraps the format string, the at-limit warn silently breaks (the visitor-`field.name() == "message"` guard fails). Deferred-work entry; revisit on async-opcua upgrade.

**Why it shipped this way:** the alternative (forking async-opcua to add a session-rejected callback, or contributing the callback upstream) is materially larger work for the same Phase A goal. Tracked as a follow-up upstream feature request.

### 5. Doctest baseline rot now in its third epic

56 doctest failures predate Epic 6 and were carried forward in Epic 6's retrospective with the same "out of scope per scope-discipline rule" disposition. Epic 7 carried them forward again. Three epics, no progress, compounding entropy.

The longer this sits, the harder the cleanup. Action item below escalates from "carry forward" to "schedule it as a defined story".

## Lessons Learned

1. **The two-event correlation pattern works, but document the silent-degradation footgun *in code*, not just docs.** Story 7-2 + 7-3 both rely on async-opcua's `info!` accept event being visible. `docs/security.md` mentions the requirement; `src/main.rs::initialise_tracing` does not enforce or warn on it. Action item below adds the startup warn.
2. **The user's "prefer one pass too much" preference explicitly endorses multi-iteration security reviews.** Save this as a load-bearing project value — multi-iteration reviews on security-touching stories are the *default*, not a quality concern. Pre-stage the cost in security-story planning rather than treating it as an overrun.
3. **Layer-as-control-plane is powerful but brittle.** When using a `tracing-subscriber::Layer` to bridge a missing upstream callback (`AtLimitAcceptLayer`), document the upstream emission-style assumptions inline + open a "request upstream callback" follow-up issue. Epic 7 did this for Story 7-3 (deferred-work entry + GitHub issue #90 follow-up — needs an analogous upstream-feature-request issue for the session-rejected event).
4. **HMAC + constant-time-compare is the new default for authentication.** Story 7-2's pass-3 refactor is the simplest threat-model upgrade in the epic. Future stories adding new credential-comparing code paths must use the same pattern; deviation requires an explicit "why" in the spec.
5. **`config.example.toml`-style "preserve operator's real-world shape" templates are landmines.** Story 7-1's `config.example.toml` shipped real IDs because the workflow was "copy the existing file aside". The lesson: when copying any production-shaped artifact into a committed reference, sanitise as a checklist step + grep-verify before commit. Adversarial review caught it; the next time it might not.
6. **Carry-forward debt is a forcing function for an Epic 8 spec refinement.** Issues #88 (per-IP rate limiting) and #89 (subscription / message-size limits) become materially more relevant once subscription clients can spawn long-lived multi-monitored-item sessions. Updating Epic 8's spec with these constraints **before** writing 8-1's story spec is the user-confirmed plan (Q4 = B).
7. **The `private_key_path` `..` / absolute-path traversal vector was not in the original spec.** Story 7-2's pass-1 review found it. The lesson: when a story introduces a new file-system-resolved path (any path that gets `Path::join`-ed into a base directory), include path-traversal rejection as a baseline AC. Promote to a project-wide spec template change.

## Significant Discoveries

### Discovery 1: NFR12 source-IP correlation degrades silently under reduced log levels

Both Story 7-2's `opcua_auth_failed` warn and Story 7-3's `opcua_session_count_at_limit` warn rely on async-opcua's `info!`-level `Accept new connection from {addr}` event being visible. If an operator sets `OPCGW_LOG_LEVEL=warn` (or higher) to dampen log volume, the global `LevelFilter` filters out the accept event before it reaches any layer or appender — and:

- The auth-failed warn loses its preceding correlation line (operators can't grep-by-timestamp to find source IP).
- The `AtLimitAcceptLayer`-emitted at-limit warn loses its `source_ip` field (the layer never sees the filtered event).

Documented in `docs/security.md`; **not enforced in code**. No error message; no warning; the field just goes missing. Exactly when an incident-response operator most needs the audit trail (high-volume reduced-noise mode), the trail breaks.

**Impact:** Affects every Epic 8 session — subscription clients are sessions, so this footgun extends to Phase B. Decided action below: emit a one-shot startup warn (Q3 = b).

**Related upstream feature request worth filing:** ask async-opcua to extend `AuthManager` with peer `SocketAddr` (closes the underlying gap, not just the symptom). Tracked at #85 / Story-7-2 deferral; **add a new issue specifically for the session-rejected callback** so 7-3's brittleness has its own tracker.

### Discovery 2: Adversarial multi-layer review is the only way the `config.example.toml` leak gets caught

Story 7-1's `config.example.toml` was the operator's real `config.toml` moved aside two days earlier. A single-reviewer pass (or even a single-layer-multi-iteration pass) would not have flagged it — the file was *correctly* preserving "real-world shape" per the spec. Only the cross-cutting check of "does this committed file contain operator-identifying production data?" surfaced it.

Blind Hunter + Edge Case Hunter combo caught it. The Acceptance Auditor would not have (the AC said "preserve real-world shape", the file did).

**Impact:** confirms the multi-layer adversarial review framework is necessary for security-domain stories. A single-layer review on a security-hardening epic would have shipped real credentials in a `.example` file.

### Discovery 3: Per-IP rate limiting becomes structurally important for Epic 8

Story 7-3's flat global `max_connections` cap defends against connection-flood DoS but not against a single authenticated client spawning 10 sessions × 100 monitored items × 10 messages/second per item. Once Epic 8 lands subscriptions, the per-IP throttle (issue #88) is no longer "nice to have, defer to Phase B" — it's a load-shaping requirement.

**Impact:** Epic 8 spec must surface message-rate / monitored-item caps as ACs in 8-2 (subscription support). The per-IP token-bucket layer can wait, but the configurability must be there from 8-2's first commit.
