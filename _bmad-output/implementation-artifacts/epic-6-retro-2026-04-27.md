## Executive Summary

Epic 6 closed the observability gap that Epic 5's retrospective explicitly flagged as a production blocker. Three stories delivered structured logging end-to-end (`tracing` with microsecond UTC timestamps and `request_id` correlation), configurable verbosity (CLI / env / TOML precedence chain, per-module file appenders untouched), and a remote-diagnostic operation taxonomy (`chirpstack_outage`, `error_spike`, `staleness_transition`, `health_metric_read`, `storage_query` with budget warnings on `opc_ua_read` / `storage_query` / `batch_write`).

Code review surfaced two large mid-epic decisions — clippy cleanup as an AC and the `Mutex<HashMap>` → `DashMap` migration on the OPC UA staleness cache — both adopted, both shipped clean. The third code-review iteration produced 5 helper extractions (`maybe_emit_error_spike`, `maybe_emit_chirpstack_outage`, `validate_bool_metric_value`, `classify_and_log_grpc_error`, `format_last_successful_poll`) that converted the synthetic-test gap from Story 6-3's iter-1 review into production-driven test coverage.

The user manual was rewritten to v2.0 covering Epics 2-6 (the v1.0 manual documented placeholder configuration keys that did not match the implementation). Two new CLAUDE.md rules were added during the work: per-story commit discipline + push-after-retrospective, and a code-review-loop discipline that requires explicit acceptance of every remaining HIGH/MEDIUM finding before a story can flip to `done`.

Epic 5's planned "ship Epic 6 before any deployment" gate is now satisfied. Next up: Epic 7 (Security Hardening) — credential management, OPC UA security endpoints, connection limiting.

## Stories Completed

### Story 6-1: Comprehensive Logging Infrastructure ✅

**Delivered:**

- Migrated console output to stderr; root file appender (`opc_ua_gw.log`) and four per-module daily-rolling appenders (`chirpstack.log`, `opc_ua.log`, `storage.log`, `config.log`) all formatted with microsecond UTC timestamps via `ChronoUtc`.
- `LoggingConfig { dir, level }` struct in `AppConfig`; figment env-overrides under `__`-split (project-wide convention now established — the original doc-comment claim that `OPCGW_CHIRPSTACK__SERVER_ADDRESS` worked was actually a docs lie until this fix).
- Per-OPC-UA-read `info_span!` with a fresh `request_id` UUID v4; downstream `storage_query`, `staleness_check`, and error logs auto-tag without threading.
- Storage-query timing via a new `StorageOpLog` Drop guard wrapping all 21 `SqliteBackend` trait methods.
- Bench captured: p50 5 µs / p95 11 µs / p99 12 µs / max 391 µs at 1k iterations against `SqliteBackend` (cold-cache outlier; AC budget was <110 ms — comfortably under).

**Highlights:**

- Two-phase init introduced (TOML peek for `[logging]` before tracing comes up) so config-load errors reach the file appender from the very first message.
- Bulk wrapping intentionally scoped to hot paths to keep the diff reviewable.

### Story 6-2: Configurable Log Verbosity ✅

**Delivered:**

- `parse_log_level` + `resolve_log_level` with explicit precedence: CLI `-d` count > `OPCGW_LOG_LEVEL` > `OPCGW_LOGGING__LEVEL` > `[logging].level` > `info` default.
- Per-module file appenders left at TRACE regardless of the global filter, so even at `error` level operators get full per-module traces.
- `docs/logging.md` — operator-facing reference with level-semantics table, worked examples, structured-field reference, correlation-ID tracing.
- Microbench: `trace! @ ERROR level` measures **0.46 ns/iter** in release mode (single comparison + branch).

**Highlights:**

- Stayed on `LevelFilter` rather than `EnvFilter` — `OPCGW_LOG_LEVEL` is the documented contract; mixing in `RUST_LOG` directive syntax broadens API surface beyond AC#1. The `env-filter` feature is enabled in `Cargo.toml` so a future swap is one line away.
- New CLAUDE.md "Documentation Sync" rule landed during this story (added by user request mid-implementation): every commit verifies README.md matches the new behavioural surface.

### Story 6-3: Remote Diagnostics for Known Failures ✅

**Delivered:**

- Microsecond-precision timestamps (already wired in 6-1, locked down in 6-3).
- Three performance budgets with automatic `warn` upgrade: `opc_ua_read` > 100 ms, `storage_query` > 10 ms, `batch_write` > 500 ms.
- Data-anomaly logs: NULL `LastPollTimestamp` warn at startup, `staleness_boundary` debug for ±5 s of threshold, `error_spike` warn for delta ≥ 5 across consecutive cycles.
- `chirpstack_connect` diagnostic logs on every existing connection branch (no new control flow); `chirpstack_outage` warn on the first per-cycle ChirpStack failure with rfc3339-rendered `last_successful_poll`.
- Edge-case logs: `gateway_status_init` (process-wide one-shot), `chirpstack_request` for `DeadlineExceeded`/`Unavailable`/`Cancelled`, `metric_parse` for invalid boolean coercion.
- Transient-failure logs: `device_poll`, `SQLITE_BUSY` (with `sqlite_error_code` sibling field for `DatabaseBusy` / `DatabaseLocked` differentiation).
- End-to-end `request_id` correlation verified by integration test (`storage_query` precedes `staleness_check`; all read-path lines share the same UUID).

**Highlights:**

- `Mutex<HashMap>` staleness cache replaced with `DashMap` for lock-free concurrent OPC UA reads (preserves the Story 5 retrospective lock-free invariant).
- Workspace clippy cleaned (60 errors → 0) via `cargo clippy --fix` plus targeted `#![allow(dead_code)]` on scaffolded modules; the orphan doc-block on `convert_metric_to_variant` was removed.
- 5 helpers extracted in iter-3 (`maybe_emit_error_spike`, `format_last_successful_poll`, `maybe_emit_chirpstack_outage`, `validate_bool_metric_value`, `classify_and_log_grpc_error`) so the synthetic-test gap from iter-1 became production-driven coverage; net +12 tests rewritten/added.

## What Went Well

### 1. Epic 5 retrospective foresight paid off

Epic 5's retro flagged "observability gap is a critical blocker" with surgical precision. Epic 6 was scoped tight to that gap — no scope creep, no orthogonal feature work. Three stories, three weeks of elapsed effort against the spec, and the gap closed cleanly. The retrospective-driven planning model is working.

### 2. Three-iteration code review discipline

Story 6-3's review went from 44 initial findings (44 → 18 → 8 → 6 → 0 across the iterations) to a `done` flip with zero unresolved HIGH/MEDIUM items. The mid-flight CLAUDE.md "loop discipline" rule (added in this same epic) materially changed the close-out: it forced explicit acceptance of every remaining issue, which in turn drove the iter-3 helper-extraction work. Without the rule, those would have shipped as "deferred-by-default" — now they're real production-driven tests.

### 3. Mid-epic CLAUDE.md rule additions stuck

Three rules added during Epic 6, all in response to observed pain:

- **Documentation Sync** (mid-6-2) — every commit keeps README in sync. Caught the v1.0 user-manual drift before it shipped.
- **BMad Workflow Commit & Push Discipline** (mid-6-3) — per-story commits, push after retrospective, never skip the retrospective. Directly addresses the situation where 6-1, 6-2, 6-3 were all uncommitted at iter-1 review time (3-story bundle was a one-time exception).
- **Code Review & Story Validation Loop Discipline** (mid-6-3) — re-run review after patches; loop terminates only at zero HIGH/MEDIUM. Drove the iter-3 work that produced the helper extractions and improved the test suite.

These rules are project-shaping. They turn one-off pain into permanent process improvement.

### 4. The DashMap migration as a case study

The iter-1 review surfaced the `Mutex<HashMap>` staleness cache as a concurrency regression (contradicted Epic 5's lock-free architectural invariant). User picked the option-2 fix (DashMap). Implementation was 5 lines of `Cargo.toml` + `use` rewrites + a 1-line constructor change + simplification of the lock + match block to a single `last_status.insert(...)` call. Zero test failures. The lock-free architecture pattern from Epic 5 propagated cleanly into Epic 6's instrumentation layer.

### 5. Helper extraction as a test-quality fix

The iter-1 review correctly identified that 6 unit tests in `chirpstack.rs` were synthetic — they emitted their own warns and asserted on the result, never invoking production code. The user picked option 2 ("address all in iter-3"), which turned into 5 helper extractions and 12 production-driven tests. The refactor itself made the production code more testable (pure functions over `tonic::Status`, not `&self` methods on `ChirpstackPoller`). Test quality is now load-bearing, not theatre.

### 6. v2.0 user manual rewrite caught v1.0 drift

The Epic 5 retrospective's "ship before deployment" gate forced a fresh look at the user manual. v1.0 (April 20) documented `[global].log_level`, `opc_ua_endpoint`, `retention_days` as configuration keys — none of which exist. v2.0 reflects the actual TOML schema, the full `[chirpstack]` / `[opcua]` / `[command_delivery]` / `[[application]]` hierarchy, and the Epic 6 logging architecture. Without the retrospective, this drift could have shipped to operators.

## What Was Hard

### 1. Synthetic-test discovery was late

The iter-1 review caught the synthetic-test pattern — but it was in 6 separate tests across two acceptance criteria. The pattern slipped through Story 6-1 (where the tests originated) and Story 6-2 (which inherited them) before Story 6-3's review surfaced it. Earlier detection would have spread the helper-extraction work across three stories instead of bunching it at the end.

**Why it slipped:** the tests asserted on tracing-test captured output, which is a real invariant — the warn shape is correct. They just never drove the production path. The pattern reads like a test of the warn macro, which is technically validated. A reviewer not specifically looking for "does this test invoke production code?" would mark it green.

### 2. Three stories' work entangled in the same files

By iter-1 review of Story 6-3, the working tree had uncommitted code from Stories 6-1, 6-2, AND 6-3 all interleaved across `src/chirpstack.rs`, `src/main.rs`, `src/opc_ua.rs`, `src/storage/sqlite.rs`. The diff against `main` HEAD was 3462 lines blending three stories' deliverables. Splitting cleanly was effectively impossible — the new commit-discipline rule was a direct response.

**Net cost:** the Story 6-3 commit ended up bundling 6-1 + 6-2 + Epic 5 retro as a one-time exception. Per-story rollback is lost for those three. Going forward the rule prevents recurrence.

### 3. Two spec-drift surprises emerged from iter-1 patches

Iter-1 made two reasonable-looking patches (`timeout_set=false` instead of `timeout_secs`, `SQLITE_LOCKED` as a separate label) that the iter-2 Acceptance Auditor flagged as drifting from the AC literal text. Both took explicit user decisions in iter-3 to resolve (D-AC1 → revert to `timeout_secs=0u64`; D-AC7 → merge to canonical `SQLITE_BUSY` with new `sqlite_error_code` sibling field).

**Why it happened:** iter-1 was triaging 44 findings under "batch-apply non-controversial patches" and the auditor's drift-detection wasn't run until iter-2. The loop-discipline rule (re-run review after patches) caught it; without it the patches would have shipped as silent spec drift.

### 4. The clippy cleanup was bigger than expected

AC#10 demanded `cargo clippy --all-targets -- -D warnings` clean. Iter-1's first run showed 60 errors. `cargo clippy --fix` handled most (25-ish), but the remainder needed targeted `#![allow(dead_code)]` annotations on scaffolded modules (`storage/{mod,memory,types,sqlite}.rs`, `command_validation.rs`) and per-item allows on legacy entry points (`AppConfig::new`, `get_device_name`, etc.). The scaffolding-vs-dead-code judgment is intrinsically subjective; the pragmatic choice was module-level allow with documented rationale.

**Hidden cost:** masking dead code at module scope means future genuine dead code in those modules will not surface as a warning. The Story 6-2 review already flagged a similar concern about `LoggingConfig` and chose a per-item allow for the same reason. This is consistent project-wide; revisit when the storage scaffold is fully wired.

### 5. Doctest baseline rot

The 56 doctest failures predate Story 6-3 and were repeatedly noted as "out of scope per scope-discipline rule". They will continue to compound until someone takes them on as a dedicated cleanup. The longer they sit, the harder the cleanup.

## Lessons Learned

1. **Retrospective foresight is a force multiplier.** Epic 5's retro pre-staged Epic 6 with such precision that scope discipline was easy. Carry that forward into Epic 7 — pre-stage the scope from this retrospective explicitly.
2. **"Loop until only LOW remains" is the right close-out gate.** It surfaces real spec drift (the iter-2 surprises), drives helper extraction (the iter-3 work), and forces explicit acceptance of every deferral. Without it, deferrals accumulate silently.
3. **Synthetic-test detection deserves an explicit reviewer prompt.** A test that captures log output is technically valid; a test that drives production code is robust. The distinction must be load-bearing in the reviewer's instructions.
4. **Each story commits on its own.** The 6-1/6-2/6-3 entanglement made every review iteration noisier than it needed to be. The new CLAUDE.md rule prevents recurrence — Epic 7 must follow it from Story 7-1 onwards.
5. **`cargo clippy --fix` is a real time-saver, but not a substitute for judgment.** It auto-fixed ~75% of the 60 errors. The remaining 25% required architecture-aware decisions (module-level vs per-item allows, refactor vs annotate). Allocate time for both.
6. **Helper extraction makes test quality load-bearing.** Pure functions over data are testable; `&self` methods on a fully-constructed struct often aren't. The 5 helpers shipped in iter-3 are the cleanest part of Epic 6.
7. **Documentation drift compounds silently.** v1.0 user manual had wrong configuration keys for over a week before anyone caught it. The new Documentation Sync rule treats README as load-bearing; the manual now sits one tier below that. Future epics should bump the manual in lockstep, not in a "before deployment" panic at the end.

## Significant Discoveries

### Discovery 1: Code-review loop discipline is a process upgrade

Three iterations on Story 6-3 produced material improvements — helper extractions, spec-drift catches, test-quality fixes. The iteration count alone is not the lesson; the lesson is that **explicit-accept-or-fix** at each iteration boundary forces engagement with every HIGH/MEDIUM finding. Drift items that would have shipped under a "batch-deferred" model were caught.

**Impact:** apply to every code review going forward. The CLAUDE.md rule formalises it.

### Discovery 2: Helper extraction reveals architectural intent

The 5 helpers extracted in iter-3 (`maybe_emit_error_spike`, etc.) didn't just fix a test gap — they made the production code's architectural intent explicit. The error-spike threshold became a constant; the gRPC error classification became an enum; the rfc3339 rendering became a named function. Future readers (and reviewers) get a clearer signal.

**Impact:** when a test feels synthetic, the right fix is often a refactor that makes the production code more testable. Not adding more tests around a bad shape; reshaping the code.

### Discovery 3: Documentation drift is silent until forced

The v1.0 user manual was wrong on day one (placeholder configuration keys that didn't match `config.toml`). It would have remained wrong indefinitely without the retrospective forcing a review. The new Documentation Sync rule treats README as the high-frequency check; the v2.0 manual rewrite establishes that user-facing documentation drifts faster than code and needs its own discipline.

**Impact:** Epic 7 must include a documentation update step in the retrospective gate, not as a pre-deployment scramble.
