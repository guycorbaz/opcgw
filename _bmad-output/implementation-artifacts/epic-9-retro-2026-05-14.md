## 2. WIN: Iter-3 over-reviewing doctrine validated across 6 stories

**User-elected biggest win.**

The `feedback_iter3_validation` memory pattern — same-LLM 3-iteration code review with the doctrine that iter-2 catches load-bearing HIGH-REGs in iter-1's own patches, and iter-3 surfaces only LOW spec-hygiene findings — was validated to a 6-story precedent across this epic:

| Story | Iter-1 raw | Iter-2 HIGH-REGs surfaced | Iter-3 HIGH-REGs surfaced | Loop termination |
|---|---|---|---|---|
| **4-4** (Phase A carry-forward, retro'd during Epic 9) | 9 | 0 (iter-2 was clean) | 1 HIGH (D2 retry/delay default violating AC#4) | iter-3 condition 1 (zero open) |
| **9-4** | 33 patches | 4 HIGH-REGs (P25-P28 iter-1's own patches) | 3 HIGH-REGs (P40-P42 iter-2's own patches) | iter-3 condition 2 (only LOW) |
| **9-5** | 22 patches | 2 HIGH-REGs (double-warn + tests #35/#36 inadequate) | 0 HIGH-REGs | iter-3 condition 2 |
| **9-6** | 22 raw → 10 patches | 2 HIGH-REGs (B-H6 asymmetric check + Axum JsonRejection audit-bypass) | 0 HIGH-REGs (12 LOW spec-hygiene) | iter-3 condition 2 |
| **9-7** | 24 patches | 2 HIGH-REGs (P25 tautological subscription test + P26 topology_device_diff/classify_diff disagreement) | 1 MED (P42 exhaustive cross-variant match) | iter-3 condition 2 |
| **9-8** | 33 raw → 10 patches | 1 HIGH-REG (**convergent across Blind + Edge — P1+P2 retry loop on Phase 4 failure**) | 0 HIGH-REGs (12 LOW) | iter-3 condition 2 |

**Critical empirical validation:** Story 9-8 iter-2's HIGH-REG was a load-bearing finding — without the iter-2 catch, every Phase 2/3/4 partial failure would have permanently wedged the gateway via the P1+P2 retry loop. A single-pass review would have shipped this bug. The iter-3 over-reviewing doctrine is therefore not just a stylistic preference; it has a documented track record of catching production-grade bugs that would not surface in iter-1.

**Cumulative impact across Epic 9:**
- Stories 9-4 through 9-8 each ran 3-iteration code reviews: 23 + 6 + 7 = ~30-35 patches per story average across the 5 review-loop stories = ~150+ patches caught by adversarial review that would have otherwise shipped (most LOW, but the HIGH-REGs would have been production hazards).
- Memory pattern is now sufficiently precedented (6 stories) to call it a **doctrine** rather than a hypothesis.

**Why this works (synthesis):** The adversarial layers (Blind Hunter / Edge Case Hunter / Acceptance Auditor) catch different concern classes — Blind catches diff-only logic bugs, Edge catches boundary cases the implementer missed, Auditor catches spec drift. Same-LLM runs over-classify on iter-2 (Blind especially), but the convergent findings across 2-3 layers are highly reliable signals of real bugs. Iter-3 is the natural same-LLM termination because: (a) the over-classification settles into LOW-only spec-hygiene findings, (b) the patch-introduces-new-regression rate drops sharply between iter-2 and iter-3, and (c) the cost-benefit shifts toward shipping at that point.

**Action item:** Continue applying this doctrine through Phase C. Consider running iter-2 or iter-3 on a **different LLM** at least once to compare same-LLM vs cross-LLM convergence — the CLAUDE.md doctrine recommends cross-LLM but it's never been empirically tested in this codebase.

## 3. LESSON: Commit at status transitions — never re-flake the Story-8-2 slip

**User-elected biggest lesson.**

Epic 9 saw mixed commit discipline across its 9 stories:

| Story | Commits | Pattern | Status discipline |
|---|---|---|---|
| 9-0 | 2 commits (spike + review-fixes) | Per-state transition | ✅ Clean |
| 9-1 | (single commit pre-epic) | n/a | n/a |
| 9-2 | (single commit pre-epic) | n/a | n/a |
| 9-3 | (single commit pre-epic) | n/a | n/a |
| 9-4 | 4 commits (ready-for-dev / validation / impl / review) | Per-transition | ✅ Clean |
| **9-5** | **1 single combined commit (2b95553)** | **Implementation + Code Review combined** | ❌ **Slip — Story-8-2 anti-pattern** |
| 9-6 | 4 commits (ready-for-dev / validation / impl / review) | Per-transition | ✅ Clean |
| 9-7 | (per-transition) | Per-transition | ✅ Clean |
| **9-8** | **1 single combined commit (8a5f450)** | **Implementation + Code Review combined** | ❌ **Slip — Story-8-2 anti-pattern (acknowledged in commit message)** |

**Root cause analysis:** When the dev workflow (`bmad-dev-story`) flips the story status from `in-progress` to `review` in the spec file + sprint-status, it does NOT automatically commit. The user has to remember to `git commit` manually before invoking `bmad-code-review`. When the user is operating in a tight feedback loop with the assistant (as in this session for Story 9-8), the "review" status transition can fly by without a commit checkpoint — leading to the combined-commit anti-pattern that CLAUDE.md explicitly flags.

**The CLAUDE.md rule (reproduced for emphasis):**
> "Implementation always lands in its own 'Implementation Complete' commit *before* any review-fix commit, even when the review concludes the same day. Story 8-2's combined 'implementation + review' single commit (cb206d6, 2026-04-30) is the precedent the Epic 8 retrospective flagged as a slip; the rule above clarifies it."

Epic 9 re-flaked this slip twice (9-5, 9-8), with the 9-8 instance specifically acknowledged in the commit message body. That's not ideal — the rule should hold without per-commit acknowledgment.

**Action items for Phase C:**

1. **AI1: Add a pre-commit gate or workflow checkpoint** that prevents `bmad-code-review` from running when the current story spec's Status is `review` AND the working tree has uncommitted changes to story-spec/sprint-status files. The gate would emit: "Story is in `review` status but Implementation Complete commit not yet made — commit first or pass `--allow-combined-commit` to override." **Owner: bmad-builder + workflow-builder collaboration.**

2. **AI2: Update the `bmad-dev-story` workflow's Step 9 ("Story completion and mark for review")** to explicitly suggest a commit before exiting. Current Step 9 saves the spec file and updates sprint-status but doesn't prompt for a commit. **Owner: bmad-builder.**

3. **AI3: Make the per-transition commit pattern (Story 9-4/9-6 shape) the canonical form** documented in CLAUDE.md, with the single-combined-commit pattern explicitly marked as "acceptable only when the implementation was already committed at a prior checkpoint (e.g., during multi-session work)." **Owner: docs maintainer (Guy).**

## 4. Other significant lessons (not user-elected but worth capturing)

### 4.1 The 9-0 spike pinned 3 load-bearing technical decisions that prevented production bugs

The Q1/Q2/Q3 spike verdicts (run-time add-path semantics; Q2 set_values mitigation for delete; Q3 lock-hold envelope) all became load-bearing in 9-7/9-8 implementations. The **empirical refinement of Q2** (discovered in Story 9-8 iter-1 test failure: async-opcua's default `MonitoredItemFilter::None` compares only `value.value` not `value.status`, so `Variant::Empty` sentinel is required to force the notification) was the most consequential finding of the entire epic — the 9-0 spike report's prose was directionally correct but underspecified.

**Action item AI4:** Future spike reports should validate every prose-level recommendation with an actual integration test against the API. The 9-0 spike's "use `manager.set_attributes(BadNodeIdUnknown)`" prose was technically wrong (should be `set_values` with `Variant::Empty + BadNodeIdUnknown` status); the 9-8 implementation discovered this at run-time. **Owner: spike author + reviewer roles.**

### 4.2 The library-wrap-not-fork pattern scaled across 7 stories

Stories 7-2 (auth) + 7-3 (connection limiting) + 8-3 (HistoryRead) + 9-0 (runtime mutation pattern) + 9-7 (config_reload listener) + 9-8 (apply pass) all extended async-opcua via composition rather than forking. The discipline held even when async-opcua's API surface was constraining (Task 6 option-(b) stub for the absent `remove_read_callback` is the most extreme case — option-(b) doc-+-FR is a legitimate scaling path for the wrap pattern when upstream lacks a hook).

### 4.3 AC#9 strict-zero file invariants — a powerful coordination tool

Each Epic 9 story explicitly listed files that MUST NOT be modified (`src/web/auth.rs`, `src/opc_ua_auth.rs`, `src/opc_ua_session_monitor.rs`, `src/security*.rs`, etc.). The discipline held: across 9-4 → 9-8, **zero production code in these files changed**, despite each story being substantial. This invariant was preserved by:
- Listing the files explicitly in each story spec
- Verifying via `git diff HEAD --stat` after each implementation
- Auditor reviewer layer re-verifying at each review iteration

This is an effective coordination tool for keeping security-relevant code stable while ancillary features evolve. **Recommend continuing for Phase C.**

### 4.4 Inline test-harness duplication reached the 3-file threshold

`tests/opcua_subscription_spike.rs` + `tests/opcua_dynamic_address_space_spike.rs` + `tests/opcua_dynamic_address_space_apply.rs` all inline-duplicate the `setup_dyn_test_server` / `open_session` / `subscribe_one` / `HeldSession` harness (~300 LOC each = ~900 LOC of triplicated code). This was a known #102 deferral throughout Epic 9; **at 3 sites the cost of NOT extracting now exceeds the cost of extraction.**

**Action item AI5:** Issue #102 (extract `tests/common/opcua.rs`) is the natural first infrastructure story for Phase C. Recommend addressing before Epic 10 implementation if not as part of Epic A. **Owner: Guy decides priority.**

## 5. Production deployment blocker: Issue #108

**Status: BLOCKING production deployment. Carries forward to Epic A.**

Issue #108 (storage payload-less MetricType) was discovered during Story 9-3 review iter-1 (2026-05-03). The `MetricType` enum in `src/storage/types.rs` has variants like `Float`, `Int`, `Bool`, `String` that carry NO payload — they're just type tags. So `metric_values` storage rows have `value == data_type` (e.g. "Float" as a string), not the actual measurement.

**Affected stories:** Stories 9-2 (dashboard `GET /api/status` reads gateway metrics correctly because Gateway health uses a different storage path) and 9-3 (live metric values display — surface-correct but data-incorrect) are **shipped with a known data-incorrectness bug** until #108 lands. Stories 5-1 (OPC UA Server SqliteBackend), 5-2 (stale data detection), 8-3 (HistoryRead) all inherit the bug — they correctly route DataValues through the storage layer, but the storage layer returns the type-name string instead of the actual value.

**Why it's not yet fixed in Epic 9:** Fix requires an Epic-A-scale storage trait refactor — every variant must carry payload (`Float(f64)`, `Int(i64)`, `Bool(bool)`, `String(String)`), every UPSERT call must serialize correctly, every READ call must deserialize correctly, all 5 epics' metric-handling code paths must be updated. The blast radius made it inappropriate as a mid-Epic-9 fix.

**Action item AI6 (CRITICAL):** Open **Epic A — Storage Payload Migration** as the immediate next epic after Phase B closure. Defining its scope:
- Storage trait extension: `MetricType` carries payload variants
- SQLite schema migration (v007): `metric_values.value` interpreted per data_type
- All UPSERT call sites updated (poller in `src/chirpstack.rs`, restore in `src/main.rs`, history append in storage layer)
- All READ call sites updated (`OpcUa::get_value` in `src/opc_ua.rs`, `OpcgwHistoryNodeManagerImpl::history_read_raw_modified` in `src/opc_ua_history.rs`, dashboard handler in `src/web/api.rs`)
- Story 9-2 + 9-3 regression tests added to verify actual value round-trip (currently absent — that's how #108 escaped review)

**Owner: Guy.** **Priority: blocks any production deployment of Epic 1-9 functionality.**
